# PMF 验证领域画像

Product-Market Fit（产品市场匹配度）验证是产品从 0 到 1 阶段最关键的决策点。
本领域画像为 Ripple 引擎在 PMF 验证任务中提供认知基础和方法论框架。

## 1. PMF 方法论概述

### 1.1 Sean Ellis 测试

Sean Ellis 提出的核心问题："如果明天不能再使用这个产品，你会有多失望？"

- **≥40% 回答 "非常失望"** → 强 PMF 信号
- **25-39%** → 有潜力但需聚焦
- **15-24%** → 弱信号，需显著调整
- **<15%** → PMF 不足

Sean Ellis 基准是认知锚点（cognitive anchor），不是精确阈值。在 Agent 模拟语境中，
等价问题变为："模拟用户群体中，有多大比例表现出'离不开'的行为信号？"

### 1.2 Superhuman 四步验证法

Superhuman CEO Rahul Vohra 在 Sean Ellis 基础上扩展的系统方法：

1. **细分受众**：找到"非常失望"比例最高的细分群体
2. **理解价值**：分析该群体认为产品核心价值是什么
3. **改进路线图**：基于"略微失望"群体的反馈制定优先级
4. **追踪趋势**：持续追踪 PMF 分数变化趋势

在 Agent 模拟中，对应操作为：
- INIT 阶段识别最可能共鸣的 Sea/Star Agent 细分群体
- RIPPLE 阶段观察哪些群体表现出强需求信号
- DELIBERATE 阶段由 Tribunal 从多角度评估细分群体价值

### 1.3 4P 框架（Product-Market Fit Pyramid）

Dan Olsen 提出的 PMF 金字塔，从下到上：

1. **目标用户（Target Customer）**：谁是产品的核心受众？
2. **未满足需求（Underserved Needs）**：他们有什么痛点？
3. **价值主张（Value Proposition）**：产品如何解决这些痛点？
4. **功能集（Feature Set）**：哪些功能支撑价值主张？
5. **用户体验（UX）**：功能的实现质量如何？

在 Agent 模拟中，OmniscientAgent 在 INIT 阶段解析产品的 4P 定位，
Star/Sea Agent 的行为反应隐含对每一层的验证。

## 2. Agent-Native PMF 验证原理

### 2.1 与传统 PMF 验证的区别

| 维度 | 传统方法 | Agent-Native 方法 |
|------|----------|-------------------|
| 数据来源 | 真实用户调查/行为数据 | LLM Agent 模拟行为 |
| 时间成本 | 数周到数月 | 数分钟（模拟时间） |
| 适用阶段 | 产品已上线 | 概念/原型阶段即可 |
| 输出性质 | 统计结论 | 预判结论（需真实验证） |
| 成本 | 获客+运营成本 | LLM 调用成本 |

**核心定位**：Agent-Native PMF 验证是"预判"而非"证明"。它帮助团队在投入真实资源前，
快速筛选方案、识别盲点、生成假设——最终仍需真实市场数据验证。

### 2.2 双层验证架构

PMF 验证使用两层互补的验证机制：

**第一层：微观传播模拟（RIPPLE Phases）**
- Star Agent（KOL/关键用户）对产品信息的反应
- Sea Agent（普通用户群体）的群体行为模式
- 观察：需求共鸣信号、传播意愿、采纳障碍

**第二层：宏观合议辩论（DELIBERATE Phase）**
- Tribunal Agent 从多个专家视角评估证据
- 刻意引入对抗性观点（Devil's Advocate）
- 收敛为结构化评分和等级判定

两层之间的一致/分歧是重要信号：
- 一致 → 结论置信度高
- 分歧 → 需要关注分歧原因，可能揭示复杂性

### 2.3 渠道无关原则

PMF 评估的核心逻辑是渠道无关的：无论产品通过社交媒体、电商、线下还是
其他渠道触达用户，需求共鸣、竞争差异化、采纳摩擦等维度的评估框架不变。

渠道特性通过 `channels/` 和 `platforms/` profile 注入，影响的是
传播动力学参数而非评估框架。

## 3. PMF 评估核心原则

### 3.1 行为信号优于态度声明

在模拟中，应重视 Agent 的"行为"（传播、采纳、复访）而非
单纯的"态度"（说好/说不好）。真实 PMF 的标志是行为改变，不是口头认可。

### 3.2 反对过度乐观

LLM Agent 天然具有"讨好"倾向（sycophancy bias），在 PMF 评估中尤其危险。
必须通过以下机制对抗：

- Tribunal 中必须有 Devil's Advocate 角色
- 评分卡每个维度独立评判，不允许"一好遮百丑"
- OmniscientAgent 在 SYNTHESIZE 阶段需交叉验证传播层与合议层的一致性
- 置信度评估必须考虑数据充分性

### 3.3 方向明确的评分量表

所有评分维度统一为"1=最差，5=最好"的单调方向，避免歧义：
- 不使用"高=好或坏取决于情境"的维度
- 每个分数有明确的行为锚点（behavioral anchor）
- 评分反映模拟中观察到的行为证据，不是 LLM 的主观判断

### 3.4 输出责任声明

每份 PMF 评估报告必须包含责任声明：
"本等级基于 Agent 模拟，仅供参考。真实 PMF 验证需要市场数据支撑。"

PMF 等级不构成投资建议或经营决策依据。

## 4. 评估中的关键观察维度

在 RIPPLE 传播模拟中，PMF 验证关注的核心信号：

### 4.1 需求共鸣信号
- Star/Sea Agent 对产品描述的反应强度
- 自发追问和深入了解意愿
- "这正是我需要的" vs "有点意思但不急" vs "跟我无关"

### 4.2 传播意愿信号
- Agent 是否愿意向他人推荐
- 推荐时的叙事方式（功能介绍 vs 情感推荐 vs 被动转发）
- 传播链深度和广度

### 4.3 竞争替代信号
- Agent 提到的替代方案或竞品
- 切换意愿（会不会继续用现有方案）
- 差异化认知的清晰度

### 4.4 采纳障碍信号
- 价格/学习成本/信任问题
- "想用但是..."后面的内容是核心洞察
- 障碍的可解决程度

### 4.5 持续价值信号
- 首次使用后的持续兴趣
- 复访/复购/持续关注意愿
- 是否产生依赖或习惯养成预期

## 5. 多方案对比（类 AB 测试）

PMF 验证支持同时评估多个产品方案（variants），通过 ensemble 运行
获得每个方案的独立评分，并在最终报告中提供方案间对比。

对比原则：
- 每个方案独立评估，不互相影响
- 报告中位数差异，不做统计显著性声明
- 如果差异小于一个分数等级，标注"差异不显著"
- 推荐最优方案时必须说明推荐理由和局限性
